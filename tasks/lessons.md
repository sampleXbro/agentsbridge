# Lessons Learned

- **Projected agent skill fixtures must use the real `x-agentsmesh-*` contract**: A Gemini branch-coverage test failed because it used a non-existent `x-projected-agent: true` flag and expected the importer to treat it as a projected agent. Root cause: the fixture invented metadata instead of matching `serializeProjectedAgentSkill()` output (`x-agentsmesh-kind: agent` plus `x-agentsmesh-name` or an `am-agent-*` directory). Rule: when testing projected agent import paths, build fixtures from the documented/generated `x-agentsmesh-*` frontmatter shape and assert the exact canonical `.agentsmesh/agents/<name>.md` destination.

- **Do not enumerate supported targets on overview pages**: The homepage still listed every currently supported built-in tool after the user pointed out the list is growing. Root cause: I kept a marketing confidence signal as a hardcoded target enumeration instead of linking to the generated support matrix, which is the documented source of truth. Rule: on the homepage, CLI overview, and other broad overview pages, use generic coverage language and link to `reference/supported-tools`; only the support matrix should enumerate per-target support.

- **Do not put escaped newlines in plain `rg` regex patterns**: A structural check used `rg -n "hero:\n|tagline:"`, and ripgrep failed because newline matching requires multiline mode. Root cause: I mixed a multiline regex into a normal `rg` search while doing quick verification. Rule: split structural searches into separate literal patterns, or use `rg -U` only when a true multiline match is required.

- **Starlight hero action icons must be schema-valid, not just valid card icons**: `pnpm --dir website build` failed after I used `icon: sync` in a homepage hero action even though `sync` works in Starlight cards. Root cause: I assumed card icon names and hero action icon schema accept the same values. Rule: for Starlight `hero.actions[*].icon`, reuse icons already proven in hero actions or omit the icon, then run the website build before considering homepage metadata changes complete.

- **Do not trust auxiliary review scripts until they execute cleanly**: The code-review skill's `code_quality_checker.py` could not run in this environment (`python` was unavailable, and `python3` hit a syntax error in the helper script), so it produced no usable architecture evidence. Root cause: I treated an external helper as a reliable scanner before proving the local runtime and script were healthy. Rule: run repo-native gates first, and when auxiliary tooling fails before analysis, fall back to direct `rg`/test/build evidence and report the helper failure separately.

- **Full-suite watch failures need readiness synchronization before edits**: `pnpm test` repeatedly failed in `tests/unit/cli/commands/watch.test.ts` waiting for `runMatrix`, while the same file passed immediately in isolation. Root cause: under full-suite scheduler load, the test wrote the new rule before the watcher was reliably past its initial generation/readiness phase. Rule: watch tests that mutate files after `runWatch()` must first wait for an initial generated artifact or log signal, then perform the edit and wait for the expected reaction.

- **Managed generated blocks must update fixture helpers that mimic generated output**: The full suite failed in `diff.test.ts` because the fixture still manually rewrote `rules/_root.md` inside the generated root contract after the contract became a protected no-rewrite block. Root cause: I updated generation and reference rewriting tests but missed a command test that synthesizes expected generated content with an old post-processing replacement. Rule: whenever generated managed-block rewrite behavior changes, search for tests that call the appender/renderer and then mutate its output, and update those fixtures before rerunning the full suite.
- **Do not use `status` as a zsh shell variable in cleanup wrappers**: An isolated global generate check passed, but the cleanup script assigned to `status`, which is read-only in zsh, so the wrapper exited before removing the temp home. Root cause: I used a common bash variable name without accounting for the active zsh shell. Rule: use neutral names like `rc` for captured exit codes in shell cleanup snippets, and verify cleanup ran when sandbox commands fail after the main operation.
- **Discarded shared outputs must not donate their drift status to the kept artifact**: `generate --check` reported Codex `AGENTS.md` as updated even though the file content matched Codex, because collision resolution kept the Codex content but merged in the discarded Windsurf result's `updated` status. Root cause: duplicate-result metadata merging was reused for conflicting AGENTS.md superset cases where only one content body is actually written. Rule: when resolving conflicting shared artifacts by choosing a preferred/richer content body, keep that chosen result's status/currentContent; only merge statuses for truly identical content.
- **Reference-style Markdown definitions are link destinations, not prose anchors**: The full suite still expected `[ref]: .agentsmesh/...` definitions to preserve canonical `.agentsmesh` paths after the generate-level demo test proved those definitions become broken generated links. Root cause: older tests classified reference definitions as reading context even though Markdown parsers and generated-link validation treat the URL portion as a clickable destination. Rule: whenever link rebasing logic handles inline Markdown destinations, include reference-style definitions (plain, titled, and angle-bracketed) in the same destination-relative contract and update edge-case tests accordingly.
- **Do not put JavaScript template literals inside double-quoted shell commands**: While writing a temporary global-mode `agentsmesh.yaml`, I used a JS template literal in a double-quoted `node -e` command, so zsh tried to execute YAML lines like `version:` before Node ran. Root cause: I forgot the existing backtick shell-expansion lesson applies to inline JavaScript too, not just `rg` patterns. Rule: for `node -e` snippets that need multiline strings, use single-quoted shell strings plus JS string arrays joined with `\\n`, or write a checked-in/temp file through a safer script path.
- **Shared generated skill artifacts cannot contain target-specific rewritten probes**: The link-rebase demo skill initially included target-native prose paths such as `.codex`/`.agents`/`.gemini` examples, and all-target generation failed because Codex and Antigravity both emit `.agents/skills/link-rebase-edge-cases/SKILL.md` with different rewritten content. Root cause: I put target-specific rewrite probes inside canonical content that must collapse to one shared artifact across multiple targets. Rule: when a canonical skill is generated into shared artifact paths, keep target-native rewrite probes in protected fenced blocks or isolate them in target-filtered fixtures, otherwise shared-output collision checks will correctly reject the generation.
- **Demo markdown links must resolve after generation, not only from canonical source**: The link-rebase demo skill initially used markdown links from a canonical skill to repo-root `README.md`/`package.json`; generation rewrote one to `README.md`, and generated-link validation failed from every target skill folder. Root cause: I mixed exploratory project-root examples with clickable markdown links that must validate from generated artifact locations. Rule: in demo fixtures that are generated across targets, keep uncertain project-root examples as inline/prose paths or add explicit expected supporting files; only use markdown destinations for links known to resolve after rebasing in every generated target.
- **Full-suite failures in dirty worktrees need attribution before patching**: `pnpm test` failed in `tests/unit/core/engine.test.ts` while verifying a link-rebaser fix, but the failures came from a pre-existing dirty change in `src/targets/projection/root-instruction-paragraph.ts` that removed the `rules/_root.md` token expected by decoration tests. Root cause: broad verification ran in a worktree with unrelated modified files in nearby generation code. Rule: when a full-suite failure appears after a scoped fix, inspect `git diff` for the failing subsystem first, attribute the failure to touched vs pre-existing changes, and avoid reverting unrelated user edits without explicit approval.
- **Do not embed backticked literals in double-quoted `rg` commands**: A search for global `.agentsmesh` expectations used a double-quoted shell pattern containing backticks, so zsh tried to execute `.agentsmesh` before `rg` ran. Root cause: I repeated a known shell quoting mistake while broadening test expectation searches. Rule: for `rg` patterns containing backticks or shell metacharacters, always use single quotes with `-F`, or split the search into simpler literal patterns.
- **Resolver fallback changes must remove obsolete guard variables**: The global link fix made relative project-root fallback unconditional, but left the old `relativeToRoot` guard variable behind, causing lint and typecheck failures. Root cause: I changed the condition without sweeping the now-dead calculation. Rule: after simplifying a branch condition, immediately remove variables that only fed the old condition and rerun lint/typecheck before broader verification.
- **Refactor destructuring must preserve const narrowing for lint**: After extracting link resolution into a helper, lint failed because the destructured `translatedPath` and `matchedPath` bindings were left as `let` even though only `resolvedBeforeTranslate` needed later reassignment. Root cause: I converted several former mutable locals into one destructuring assignment without splitting mutable and immutable bindings. Rule: after extracting a helper that returns multiple values, destructure immutable returns as `const` and create a separate `let` only for the specific value that is reassigned, then rerun `pnpm lint`.
- **Global link-rewrite tests must separate prose anchors from markdown destinations**: The global-mode alignment initially left stale tests asserting that `.agentsmesh/` should disappear from every generated artifact, which conflicted with the link-rebaser readability contract and caused broad expectation failures. Root cause: the test suite treated reading-context tokens and clickable markdown destinations as the same kind of output. Rule: when changing link formatter semantics, update generate, import, round-trip, and matrix tests by token role: preserve well-known anchors in prose/labels/quoted text, but require destination-relative paths inside markdown link destinations.
- **Strict lint gates apply to touched null checks and dirty nearby tests**: The first link-rebaser alignment pass used `!= null` in new code and left unrelated unused imports in a dirty test file, so `pnpm lint` failed after targeted behavior tests were green. Root cause: I relied on idiomatic nullish shorthand and did not run lint before moving to broader verification. Rule: when editing TypeScript in this repo, use explicit strict null comparisons (`!== null` / `!== undefined`) and rerun `pnpm lint` immediately after targeted tests, fixing any gate-blocking dirty-file imports without reverting unrelated edits.
- **Markdown relative-link contract changes must update import e2e expectations too**: After forcing markdown destinations to keep explicit relative paths, unit tests were green but `import-reference-rewrite.e2e.test.ts` still expected Gemini imported markdown to remain symmetric as `[GEMINI.md](GEMINI.md)`. Root cause: I updated core/reference unit assertions but missed the dist-backed import expectation that observes the same markdown-link contract after target import. Rule: whenever markdown destination formatting changes, rerun and update both generate and import reference-rewrite e2e suites, including target-specific root-file aliases like Gemini `GEMINI.md`.
- **Full-suite e2e CLI timeouts need headroom beyond targeted runs**: The codex full-sync import passed when reproduced directly but failed inside `pnpm test` at the shared `runCli` 30s timeout. Root cause: full-suite parallel load can push otherwise-valid dist-backed CLI flows past the helper's narrow timeout budget. Rule: keep e2e CLI helper timeouts high enough for full-suite pressure, and reproduce timeout-looking failures both targeted and under the full runner before treating them as product behavior failures.

- **Quote `rg` patterns with literal backticks safely**: I searched for a lesson containing `` `pnpm build` `` inside a double-quoted shell string, so zsh treated the backticks as command substitution and ran `pnpm build` before `rg` failed with a regex parse error. Root cause: shell metacharacters in search patterns were not isolated from shell expansion. Rule: when searching for text that contains backticks, `$`, or other shell metacharacters, wrap the pattern in single quotes or use `rg -F -- 'literal text'`.

- **Do not run `pnpm build` concurrently with dist-backed e2e tests**: A delegated targeted test run rebuilt the package while the local e2e suite was spawning `dist/cli.js`, causing noisy `ENOENT` / `MODULE_NOT_FOUND` failures even though the CLI build itself was fine. Root cause: `tsup` cleans `dist` at build start, and concurrent e2e workers can hit that gap. Rule: before running tests that execute `dist/cli.js`, ensure no subagent or local shell is running `pnpm build`; build once, then run dist-backed tests without overlapping rebuilds.

- **Importer workaround bridges must preserve supporting files, not just primary files**: I strengthened Windsurf global e2e coverage to assert skill supporting files, but the first run failed because the known global-import workaround copied only `SKILL.md` into the project-style importer path. Root cause: the workaround mirrored the old minimal fixture instead of the full generated skill tree. Rule: when a test bridges generated global files into an importer fallback path, copy the exact generated tree needed for round-trip assertions, including `references/`, `scripts/`, `templates/`, and `assets/`.

- **Regex capture groups need explicit undefined guards under strict indexed access**: `pnpm typecheck` failed in `parseMarkdownLinkDestination()` because `RegExpExecArray[1]` was treated as possibly undefined even after checking that the regex matched. Root cause: the implementation relied on regex structure instead of narrowing the captured group value itself. Rule: when reading regex capture groups in strict TypeScript, guard the specific capture (`match?.[1] !== undefined`) before calling string methods, then rerun `pnpm typecheck`.

- **Generated Markdown root mirrors must be in the output-source map, not just decoration paths**: `.cursor/AGENTS.md` still leaked `.agentsmesh/skills/post-feature-qa/` after the shared link-rebaser consolidation because the source map only associated Cursor's root rule with `AGENTS.md`, not the additional `.cursor/AGENTS.md` mirror. Root cause: decoration paths were treated as render-only metadata instead of generated artifacts that need reference rewriting. Rule: whenever a layout declares `additionalRootDecorationPaths`, add those paths to generate-side source maps and test at least one nested Markdown mirror for canonical link cleanup.

- **Nested generated Markdown links need explicit current-directory prefixes**: `.cursor/AGENTS.md` was corrected from `.agentsmesh/skills/post-feature-qa/` to `skills/post-feature-qa/`, but the user expected the verified path from `.cursor/AGENTS.md` itself: `./skills/post-feature-qa/`. Root cause: `formatLinkPathForDestination()` stripped leading `./`, and the regression only asserted "not canonical" instead of the exact target-relative spelling. Rule: for generated Markdown inside a target folder, preserve/add `./` for same-directory descendant links and assert the exact rendered path in tests.

- **Global-mode e2e expectations must derive from target descriptors, not historical path helpers**: While consolidating link/path rebasing, the global path e2e suite failed on paths like Windsurf `workflows`, Gemini `workflows`, Roo `.roo-code`, and Codex `config.toml` even though production generation now uses target descriptor layouts. Root cause: the e2e expectations had drifted from descriptor constants and from the shared rewriter's protected URI behavior. Rule: when retiring path-rewrite helpers or changing global-mode tests, update e2e assertions from the target descriptor/constants and rerun them with the e2e Vitest config after `pnpm build`.

- **Global mode requires `rewriteGeneratedPath` in the target layout to transform project paths to global paths**: When implementing Copilot global mode, I initially only added the `global` layout with paths but forgot to add `rewriteGeneratedPath` to transform `.github/` paths to `.copilot/` paths. Root cause: I assumed the path resolvers in the layout would be sufficient, but the engine uses `rewriteGeneratedPath` to transform generator output paths before writing. Rule: when implementing global mode for a target, always add `rewriteGeneratedPath` to the global layout to transform project-level paths to global paths, and test that files are generated in the correct location.
- **Create new reusable skills in `.agentsmesh/skills`, not `.agents/skills`**: I initially added the global-mode target skill under `.agents/skills`, which bypassed the repo's canonical source-of-truth model for skills. Root cause: I followed the local runtime-visible skill tree instead of the project rule that `.agentsmesh/` owns editable canonical content and generated surfaces are artifacts. Rule: when adding or updating repo-owned skills in this codebase, create them under `.agentsmesh/skills/...` first and only rely on generation to project them into target-specific artifact folders.
- **Do not invent extra files inside canonical skill directories unless the canonical skill spec supports them**: I added `agents/openai.yaml` under a canonical skill even though the documented `.agentsmesh/skills` contract only supports `SKILL.md` plus supporting files. Root cause: I copied a pattern from existing repo content without checking it against the published canonical-config spec for skills. Rule: before adding a new file shape under `.agentsmesh/skills/<name>/`, verify it is part of the documented canonical format; otherwise keep the skill to `SKILL.md` and ordinary supporting files only.
- **Canonical loader refactors must preserve the project-root entrypoint contract**: I changed `loadCanonicalFiles()` to accept only a canonical-directory path, which immediately broke extend loading, install flows, and the loader test suite because much of the repo still passes project roots and expects the function to locate `.agentsmesh/` itself. Root cause: I optimized for the new global-scope path without preserving the existing public helper contract or sweeping every caller first. Rule: when making canonical path-loading scope-aware in this repo, keep loader helpers backward-compatible with both project-root and canonical-dir inputs until every caller is explicitly migrated and covered.

- **Watch unit timing budgets may need another bump after test-runner upgrades, even when isolated runs stay green**: After upgrading Vitest from `4.0.18` to `4.1.4`, the full `pnpm test` suite started flaking in `tests/unit/cli/commands/watch.test.ts` even though the same file still passed in isolation, because the heavier full-run scheduler pushed a `runMatrix` assertion past the existing 8000 ms wait budget. Root cause: I treated the previously hardened timeout as stable across runner upgrades without revalidating the full-suite latency envelope. Rule: when test-runner or watcher-adjacent dependencies change, rerun the full suite and increase watch-unit `vi.waitFor` budgets again if needed; isolated green runs are not enough.

- **Do not batch runtime packages into a `pnpm add -D` upgrade command**: While upgrading vulnerable dependencies, I included runtime packages (`diff`, `tar`) in the dev-dependency update command and silently moved them into `devDependencies`. Root cause: I optimized for fewer commands instead of preserving the runtime/dev boundary package by package. Rule: when upgrading mixed dependency types, run separate install commands for `dependencies` and `devDependencies`, then re-open `package.json` before considering the update pass correct.

- **Prefer a runner/toolchain that already satisfies npm trusted-publishing requirements over self-upgrading npm in CI**: The release workflow failed on `npm install -g npm@latest` on `ubuntu-latest` with `Cannot find module 'promise-retry'` inside npm's own arborist dependency tree before the publish step even ran. Root cause: I treated npm self-upgrade as the default fix for trusted publishing instead of checking npm's documented minimums and selecting a runner that already bundles a compatible npm. Rule: for npm trusted publishing, first choose a GitHub-hosted Node version whose bundled npm already satisfies the current docs (for example Node 24 in the current npm examples); only add an explicit npm install when a specific known-good version is required and the runner cannot supply it directly.

- **`execSync` shell must exist on Linux CI**: A lint integration test passed `shell: '/bin/zsh'` to merge stderr; GitHub Actions `ubuntu-latest` has no `/bin/zsh` → `spawnSync /bin/zsh ENOENT`. Root cause: macOS dev shells were assumed for all runners. Rule: for shell features like `2>&1` in integration tests, use `/bin/sh` (or omit `shell` and avoid shell-only syntax), never hardcode zsh-only paths unless the job installs zsh.
- **When test fixtures grow required manifest fields, update callback expectations in the same pass**: I added `source_kind` and `features` to `InstallManifestEntry` fixtures in `tests/unit/install/install-sync.test.ts`, which made standalone TypeScript clean, but the full Vitest run still failed because the `reinstall` assertions were still expecting the old `{ name, source }` payload. Root cause: I repaired the typed fixture shape without sweeping downstream expectations that compare the same object at runtime. Rule: whenever a test helper or fixture type gains required fields, update all `toHaveBeenCalledWith` / deep-equality assertions that observe that payload before calling the pass complete.
- **Repo typecheck that excludes tests must be supplemented when touching typed test fixtures**: A user still saw `TS2741` in `tests/unit/canonical/extend-load.test.ts` even though `pnpm typecheck` was green, because `tsconfig.json` excludes `**/*.test.ts` and the failing `ResolvedExtend` fixture only existed in a test file. Root cause: I relied on the repo typecheck gate without checking whether the touched file was inside the compiler include set. Rule: whenever a fix touches typed test files in this repo, run a standalone `tsc --noEmit ... <test-file>` (or an equivalent test-aware TS config) on the affected tests, especially for editor-reported TS errors.
- **If ESLint rules target tests, the lint script must target tests too**: Warning cleanup surfaced unused vars and return-type warnings in `tests/` even though `pnpm lint` was green, because the script only ran `eslint src/` while `eslint.config.js` defined rules for `tests/**/*.ts` as well. Root cause: lint command scope drifted from the configured file scope, so test warnings accumulated outside the default verification path. Rule: whenever ESLint config includes `tests/**` or other first-class source trees, keep the `lint` script aligned with those paths and ignore only generated artifacts explicitly.
- **Target-native filename corrections still need legacy import aliases when users may already have the old file**: After switching Cline to the corrected generated filename `cline_mcp_settings.json`, the importer stopped reading historical `.cline/mcp_settings.json` files and the Cline import e2e lost canonical MCP output. Root cause: I treated the filename correction as a full replacement across all read paths instead of separating generation/stale-cleanup contracts from backward-compatible import support. Rule: when a target-native filename changes, update generation and docs to the new path, but keep importer fallbacks for the previous on-disk filename unless the migration is explicitly destructive.
- **Watch tests need per-test temp roots under coverage load**: The full coverage run started flaking in `tests/unit/cli/commands/watch.test.ts` even though the watch logic was fine, because every case reused the same temp project path and chokidar state could bleed across teardown/recreate cycles. Root cause: the test fixture optimized for convenience by keeping one shared temp directory, which is unstable for file watchers when coverage slows timing and event cleanup. Rule: any watcher test in this repo must create a fresh temp project directory per test case and not reuse a prior watched path after teardown.
- **Path-preserving import refactors must update native install scope derivation and relative-path filters together**: After preserving nested canonical import paths, native install scope still derived command picks from `basename(...)` and the Cline rule mapper still looked for `"/workflows/"`, which broke Gemini namespaced command picks and caused Cline workflows to be double-counted as rules. Root cause: I updated canonical output paths without sweeping downstream code that still assumed flattened names or absolute-style path fragments. Rule: whenever imported canonical paths change shape, audit install-scope pick derivation and every relative-path classifier for basename/leading-slash assumptions, then rerun the native install scope suites.
- **Async importer serializer changes must update every `importFileDirectory` mapper to await them**: I switched Copilot prompt import to an async placeholder serializer but left the mapper callback synchronous, which wrote a `Promise` into `writeFileAtomic` and broke the importer suite. Root cause: I changed a shared helper contract without sweeping all mapper call sites that feed `importFileDirectory`. Rule: whenever an import path starts using an async serializer, make the mapper callback `async`, await the serializer result, and rerun the touched target importer suite plus `pnpm lint`/`pnpm typecheck` before broader verification.
- **New repo docs must not be authored under an ignored path without first narrowing the ignore rule**: I created the architecture documentation set and only afterward verified git visibility, which meant the files existed on disk but were still excluded by the root `/docs` ignore policy. Root cause: I treated the presence of existing tracked docs as evidence that new docs would also be trackable and skipped `git check-ignore` up front. Rule: before adding any new documentation tree in this repo, inspect `.gitignore` and run `git check-ignore -v` on a representative target path so the docs land in a trackable location or the ignore rules are narrowed intentionally.
- **Watcher startup must await chokidar readiness before doing the initial generate**: After reorganizing `install/` and `canonical/`, the full suite still showed flaky `watch` behavior even though the isolated watch suite passed, because `runWatch()` kicked off its startup generate before chokidar finished establishing watches. Root cause: startup lock churn and the first real edit could race watcher readiness under full-suite load, producing missed rule-add events and stray self-triggered regenerations. Rule: when a watch command performs an initial reconciliation/generate, wait for the watcher `ready` event first, then run the initial generate, and recheck both the isolated watch suite and full `pnpm test`.
- **Watch-mode ignore filters must account for parent directory events, not only specific generated files**: After splitting `src/core` into subfolders, the full suite still showed flaky `watch` behavior even though the targeted watch test passed in isolation, because lock writes could surface as `.agentsmesh` directory events under heavier suite load. Root cause: the ignore helper filtered `.agentsmesh/.lock` and `.agentsmesh/.lock.tmp` but not the parent directory metadata event emitted by the watcher. Rule: whenever watch mode suppresses self-generated file churn, ignore both the exact generated files and the canonical parent directory event, then validate with the isolated watch suite and the full `pnpm test` run.
- **Metadata consolidation needs immediate lint/typecheck gates before broader verification**: I centralized built-in target metadata and rewired registry/catalog lookups, but the first pass left stale skill-dir imports and widened map key types that only surfaced in `pnpm lint` and `pnpm typecheck`. Root cause: I changed the source-of-truth layer first and relied on targeted behavior tests before re-running static analysis on the touched plumbing. Rule: after any cross-cutting metadata refactor, run lint and typecheck immediately after the first green targeted tests, and type shared lookup maps (`Map<string, ...>` or narrow helper returns) so string-key lookups do not drift into widened unions.
- **Generated root decoration must avoid the `.agentsmesh/` substring when e2e asserts clean target prose**: The reference-rewrite matrix strips fenced code blocks then requires generated root files not to contain `.agentsmesh/`; the appended AgentsMesh contract used inline `` `.agentsmesh/` ``, which always failed that guard. Root cause: decoration text was written as a path literal without considering the strict “no canonical path token in target output” contract. Rule: when appending global instructions to generated targets, phrase canonical location without the exact `.agentsmesh/` substring (e.g. “the `.agentsmesh` directory at the project root”), keep legacy paragraph detection for upgrades, and strip both old and new contract blocks on import.
- **Centralized target metadata needs direct engine assertions, not only generator coverage**: I moved root-instruction decoration into engine metadata but initially sourced Cursor's primary path from a file with no exported constant, which only surfaced as a build warning and a stale unchanged-status expectation. Root cause: generator tests still passed because they no longer owned decoration, and there was no explicit engine assertion proving Cursor's registered primary root file was the one being decorated. Rule: whenever behavior moves from target generators into engine/registry metadata, add engine-level tests for each representative target path (especially compatibility targets like Cursor) and rerun build warnings as part of the refactor gate.
- **Canonical `.agentsmesh/` content must be edited before generated target artifacts**: I updated `.agents/skills/...` and `.github/skills/...` directly when fixing the release skill, and only afterward was reminded that `.agentsmesh/` is the source of truth. Root cause: I treated generated target files as primary because they were the visible diff, instead of tracing the change back to canonical source first. Rule: when a skill/rule/command/agent exists under `.agentsmesh/`, make the edit there first and regenerate affected targets rather than hand-editing `.agents/`, `.github/`, or other generated trees.
- **Trusted publishing must stay on the tested Changesets contract with a current npm CLI**: Release automation failed with `npm error code E404` during GitHub Actions trusted publishing while the repo still used a raw `npm publish` release script and the publish workflow never upgraded npm on the runner. Root cause: the release path drifted away from the repo's intended Changesets/OIDC flow and from current npm trusted-publishing guidance. Rule: keep `package.json` on `pnpm build && changeset publish`, set explicit public publish metadata (`publishConfig`, canonical `repository.url`), and upgrade npm in the publish workflow before the changesets publish step.
- **`pnpm link --global` needs `PNPM_HOME` in that shell**: It fails with `ERR_PNPM_NO_GLOBAL_BIN_DIR` when the process environment does not define `PNPM_HOME` (and global-bin-dir), which happens in some IDE-integrated terminals that skip full shell init even if `~/.zshrc` is correct for interactive use. Root cause: pnpm only knows where to place global shims when that variable is set. Rule: document or script a no-link dev path (`pnpm build` then `pnpm ab -- <args>` / `node dist/cli.js`); for global link, run `pnpm link --global` from a terminal where `echo $PNPM_HOME` is non-empty (e.g. `source ~/.zshrc` first).
- **Manual single-item installs should persist collection scope plus pick, not file scope alone**: I initially persisted manual single-file installs at the file path itself, which meant a later sibling install from the same source folder could neither match the existing pack nor merge the manifest subset cleanly. Root cause: the persistence identity for `install --as ... <single file>` was derived from the discovered file path instead of the parent collection plus selected item name. Rule: whenever a manual install targets one file or one skill from a broader collection, persist the parent collection path and explicit `pick` so later installs from the same collection can amend the same pack and installs entry.
- **Single-file remote installs need blob URL parsing, not just tree URL parsing**: I previously fixed sync replay and assumed single-file remote installs were covered, but GitHub `blob/...` URLs still fell through to local-path parsing and failed with `Path does not exist`. Root cause: the install parser only recognized `tree` repository URLs and pinned remote ids, not file-oriented `blob` URLs. Rule: whenever install supports single-file remote inputs, parser coverage must include both directory-style `tree` URLs and file-style `blob` URLs for each supported host.
- **Sync replays must restore the saved install scope, not just the saved source path**: I fixed `install --sync` to replay the caller’s validation flags, but it still reinstalled every item under the saved folder because the replay path ignored `installs.yaml` `pick`/`features` metadata. Root cause: sync reused normal discovery and rebuilt selection from the broad path instead of reapplying the persisted manifest scope. Rule: whenever install replay persists `features` or `pick`, sync must reapply that saved scope before validation, selection, and manifest rewrite so restored packs match the original install exactly.
- **Sync replays must preserve install validation semantics unless the user explicitly forces them**: I initially implemented `install --sync` by replaying every saved install with `force: true`, which silently bypassed the invalid-item prompts and conflict checks users get on a normal install. Root cause: I treated sync as a recovery shortcut instead of a batch replay of the same install contract. Rule: when replaying persisted installs, carry through the caller’s `--force` and related validation flags rather than hardcoding a more permissive mode.
- **Install sync must replay persisted remote source formats, not just user-entered URLs**: I added `.agentsmesh/installs.yaml` replay but initially forgot that successful remote installs persist pinned sources like `github:org/repo@sha`, not the original `https://.../tree/...` URL. Root cause: sync reused the normal install parser, while the parser only understood user-facing URL/SSH/local forms and treated pinned manifest sources as local paths. Rule: whenever a workflow persists internal source identifiers for replay, the replay parser must explicitly support those stored formats and be covered by a regression test that uses the persisted value verbatim.

- **`flatFile` import mode does not model multi-file merge**: Migrating cursor's `.cursorignore` import to the descriptor's `flatFile` mode broke the `import-anatomy` e2e test because cursor merges `.cursorignore` AND `.cursorindexingignore` (and dedupes lines) into one canonical `ignore` file. `flatFile` walks `source` paths in order and writes the FIRST one verbatim — it never combines. Root cause: I assumed `flatFile` was a generic "single-file copy" without re-reading the original behavior, which already had merge semantics. Rule: before migrating an `ignore` (or any single-output) feature to `flatFile`, check if the original importer reads multiple sources and concatenates/dedupes — if so, keep it imperative or extend the runner with a `mergedFlatFile` mode. Don't trust the destination filename being singular as proof the source is too.

- **Per-target import mappers must live in a third file, never in `importer.ts`**: Migrating cline/cursor/claude-code/copilot/roo-code/antigravity/junie/gemini-cli to descriptor-driven imports established the pattern: every target with a custom `ImportEntryMapper` puts those mappers in `<target>/import-mappers.ts`, with both `<target>/index.ts` (the descriptor) and `<target>/importer.ts` (the residual imperative shell) importing FROM that file. Putting mappers in `importer.ts` and importing them from `index.ts` forms a cycle: `index.ts → importer.ts → index.ts`, and the descriptor's object literal captures `undefined` for `map:` fields because `importer.ts` is still mid-evaluation when the literal runs. Rule: when adding a target with a descriptor-driven importer, create `import-mappers.ts` even if there's only one mapper — the convention is what protects future contributors from the TDZ trap.

- **Glob-shaped paths in JSDoc/header block comments close the comment early**: Authoring `tests/unit/core/merger-edge-cases.test.ts`, my header comment included `pack checksums are populated from packs/*/pack.yaml`. The `*/` inside that path terminated the leading `/**` block comment, so esbuild parsed the rest of the file as code and reported a misleading "Unterminated string literal" at EOF; preceding intermediate errors looked like "Expected ';' but found am" and "Unexpected '<<'" at unrelated lines. Root cause: the `*/` inside the glob is indistinguishable from a comment terminator. Rule (extending the rule at line 82 to test files and header blocks): never write a literal `*/` inside any `/* ... */` comment — replace globs in prose with `each pack pack.yaml` or `glob: *.md`, or move the glob into a single-line `//` comment outside the block.

- **Top-level template literals containing left-anchored `<<<<<<<` / `=======` / `>>>>>>>` lines fail TS conflict-marker detection**: Defining a module-scope `const CONFLICTED_LOCK = \`...<<<<<<< HEAD\n=======\n>>>>>>> branch\n\`` produced TS1185 "Merge conflict marker encountered"; the identical literal as an argument inside `writeFileSync(path, \`<<<<<<<...\`)` compiles fine in pre-existing tests. Root cause: TypeScript's lexer scans for left-anchored 7-character merge-conflict marker lines and refuses to compile when one is found at module top level, even inside a template literal. Rule: when a fixture string needs literal `<<<<<<< HEAD` / `=======` / `>>>>>>>` lines, either (a) inline it inside a CallExpression argument, or (b) build it from concatenated parts (`'<' + '<<<<<< HEAD'`, `'=' + '======'`, `'>' + '>>>>>> branch'`) so no source line begins with seven repeated `<`/`=`/`>`.

- **Existence gate and "form-when-exists" are two separate contracts in the link rebaser — don't conflate them in tests**: My first existence-gating test asserted that a canonical `.agentsmesh/skills/release-manager/SKILL.md` token in backtick prose would be rewritten to `../release-manager/SKILL.md` when the target exists, and 25/75 cases failed. Root cause: `shouldPreserveAgentsMeshAnchor` (in `link-rebaser-formatting.ts:87`) deliberately preserves the canonical `.agentsmesh/...` form inside reading contexts (inline-code / quoted / bracketed / bracket-label / at-prefix / bare-prose) for human readability — only markdown link destinations get rewritten to relative. So canonical tokens have a different "form-when-exists" rule than absolute / root-relative tokens, even though the existence-gate (missing → preserved) is uniform. Rule: when testing the existence gate in isolation, assert "rewrite happened (output !== input)" rather than the exact rewritten string, OR use markdown link destinations for canonical tokens since those bypass the readability preserve. Test the form-when-exists contract separately if you want exact-string assertions.

- **Cross-platform repos must commit `.gitattributes` with `text=auto eol=lf`**: Windows CI for `agentsmesh generate --check` started returning exit 1 instead of 0 because the GitHub Actions Windows runner's `core.autocrlf=true` rewrote canonical fixture `.md` / `.json` / `.yaml` files to CRLF on checkout, while `writeFileAtomic` normalizes generated output to LF. The lock checksums then disagreed with what the second pass would write, so `--check` reported drift even though the project was actually in sync. Same root cause was already documented for `matrix-drift-guard.integration.test.ts` (lessons line 175), but without a `.gitattributes` enforcing LF the issue keeps surfacing in different fixture-driven tests. Rule: any cross-platform TS repo whose tests copy fixtures into temp dirs and then assert byte-stable generated artifacts must commit a `.gitattributes` with `* text=auto eol=lf` plus explicit text categories for `*.md`, `*.json`, `*.yaml`, etc. — relying on per-runner git config is not a contract.

- **Test walkers across project trees must skip `.agentsmeshcache` and tolerate vanishing entries on Windows**: `tests/e2e/link-destination-leak-guard.e2e.test.ts` walked the whole project dir and `statSync`-ed every entry, dying with `ENOENT` on `.agentsmeshcache` between `readdirSync` and `statSync` on Windows runners — TOCTOU between an exiting CLI process and the test walker. Even when the cache survived, walking into it was wasted work because the leak filter excluded it later anyway. Rule: any test walker over a project root must (a) explicitly skip `.agentsmesh`, `.agentsmeshcache`, `node_modules`, and `.git` at directory entry level, and (b) wrap `readdirSync` and `statSync` in try/catch so a vanishing entry mid-walk does not fail the test. Apply this even to tests that already pass on macOS — it's a Windows correctness contract.

- **Skip-on-Windows is the right answer for tests that exercise POSIX absolute paths in canonical content**: `tests/integration/generate-skill-absolute-links.integration.test.ts` constructed canonical content via `${testDir}/.agentsmesh/...`, which on Windows produces `C:\Users\RUNNER~1\...\.agentsmesh\...` — a Windows absolute path that the link rebaser intentionally preserves (`WINDOWS_ABSOLUTE_PATH` guard, lessons line 178). The test's expectation (rewrite-to-relative) contradicts the production contract on Windows, so the test must opt out via `it.skipIf(process.platform === 'win32')`. Rule: when a new test asserts that absolute paths in canonical content get rewritten, gate the absolute-path scenarios with `skipIf(process.platform === 'win32')` and keep the cross-platform variants (root-relative `/.agentsmesh/...` tokens, relative `../...` tokens) running on every runner.

- **CLI display paths must normalize separators to forward slashes**: `target.test.ts` tests failed on Windows because `path.join` produces backslash-separated paths, and `p.slice(projectRoot.length + 1)` preserves the native separator. The tests asserted forward-slash paths like `src/targets/foo-bar/constants.ts` but received `src\targets\foo-bar\constants.ts`. Root cause: display-path formatting relied on native OS separators from `node:path` without normalization. Rule: any CLI output that displays file paths to the user must normalize separators with `.replaceAll('\\', '/')` before printing, so output is consistent across platforms and tests can assert forward-slash paths unconditionally.
