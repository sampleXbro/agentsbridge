# Lessons Learned

- **`pnpm link --global` needs `PNPM_HOME` in that shell**: It fails with `ERR_PNPM_NO_GLOBAL_BIN_DIR` when the process environment does not define `PNPM_HOME` (and global-bin-dir), which happens in some IDE-integrated terminals that skip full shell init even if `~/.zshrc` is correct for interactive use. Root cause: pnpm only knows where to place global shims when that variable is set. Rule: document or script a no-link dev path (`pnpm build` then `pnpm ab -- <args>` / `node dist/cli.js`); for global link, run `pnpm link --global` from a terminal where `echo $PNPM_HOME` is non-empty (e.g. `source ~/.zshrc` first).
- **Manual single-item installs should persist collection scope plus pick, not file scope alone**: I initially persisted manual single-file installs at the file path itself, which meant a later sibling install from the same source folder could neither match the existing pack nor merge the manifest subset cleanly. Root cause: the persistence identity for `install --as ... <single file>` was derived from the discovered file path instead of the parent collection plus selected item name. Rule: whenever a manual install targets one file or one skill from a broader collection, persist the parent collection path and explicit `pick` so later installs from the same collection can amend the same pack and installs entry.
- **Single-file remote installs need blob URL parsing, not just tree URL parsing**: I previously fixed sync replay and assumed single-file remote installs were covered, but GitHub `blob/...` URLs still fell through to local-path parsing and failed with `Path does not exist`. Root cause: the install parser only recognized `tree` repository URLs and pinned remote ids, not file-oriented `blob` URLs. Rule: whenever install supports single-file remote inputs, parser coverage must include both directory-style `tree` URLs and file-style `blob` URLs for each supported host.
- **Sync replays must restore the saved install scope, not just the saved source path**: I fixed `install --sync` to replay the caller’s validation flags, but it still reinstalled every item under the saved folder because the replay path ignored `installs.yaml` `pick`/`features` metadata. Root cause: sync reused normal discovery and rebuilt selection from the broad path instead of reapplying the persisted manifest scope. Rule: whenever install replay persists `features` or `pick`, sync must reapply that saved scope before validation, selection, and manifest rewrite so restored packs match the original install exactly.
- **Sync replays must preserve install validation semantics unless the user explicitly forces them**: I initially implemented `install --sync` by replaying every saved install with `force: true`, which silently bypassed the invalid-item prompts and conflict checks users get on a normal install. Root cause: I treated sync as a recovery shortcut instead of a batch replay of the same install contract. Rule: when replaying persisted installs, carry through the caller’s `--force` and related validation flags rather than hardcoding a more permissive mode.
- **Install sync must replay persisted remote source formats, not just user-entered URLs**: I added `.agentsbridge/installs.yaml` replay but initially forgot that successful remote installs persist pinned sources like `github:org/repo@sha`, not the original `https://.../tree/...` URL. Root cause: sync reused the normal install parser, while the parser only understood user-facing URL/SSH/local forms and treated pinned manifest sources as local paths. Rule: whenever a workflow persists internal source identifiers for replay, the replay parser must explicitly support those stored formats and be covered by a regression test that uses the persisted value verbatim.
- **Install needs an explicit manual collection mode in addition to native-path inference**: I initially treated installability as something that could always be inferred from the source path shape, which failed on repos that store compatible agent content in arbitrary folders like `agents/universal`. Root cause: install discovery was coupled too tightly to target-native directory heuristics instead of the user’s explicit intent. Rule: when a user specifies what kind of collection they want to install, provide an explicit override (`--as`) that bypasses path heuristics while keeping native auto-detection as the fallback.
- **Pack coverage must include settings features, not just array resources**: I initially focused pack install verification on skills/rules/commands/agents and missed that materialized packs were silently dropping canonical `mcp`, `permissions`, `hooks`, and `ignore` during write/load/merge. Root cause: the earlier tests only asserted array-resource behavior and dry-run previews, so the pack persistence contract for settings never ran through a real install. Rule: whenever install materializes canonical data, add tests that verify every supported canonical feature survives write → load → generate, including settings files and a real non-dry-run command path.
- **Native install discovery must be staged and importer-driven**: I initially let `install` write `.agentsbridge/` into the source repo during native discovery and still depended on native folder heuristics for subtree selection. Root cause: I reused the extend/import normalization path directly instead of treating install discovery as a read-only staging step. Rule: whenever `install` needs to understand a native agent structure, import into a temporary staged copy and derive folder/file scope from importer `fromPath` results rather than mutating or heuristically traversing the source checkout.
- **Incremental pack updates must match remote source identity, not the pinned ref**: Pack reinstall logic failed to find an existing remote pack once the upstream repo resolved to a new SHA because the lookup compared the full pinned `source` string (`github:org/repo@sha`). Root cause: install metadata stores the latest pinned provenance, but incremental-update detection needs a stable repo identity across versions. Rule: when materialized installs track pinned remote provenance, match existing packs by stable remote identity (provider + repo/url without ref) and refresh the stored `source`/`version` metadata after each successful merge.
- **Live generated targets must be checked after capability changes**: I claimed Continue/Junie reference rewriting was correct based on targeted tests, but the real generated repo still leaked `.agentsbridge/skills/...` links because those targets did not actually emit any skill artifacts. Root cause: I verified synthetic round trips without checking the live target outputs against the actual configured feature set. Rule: whenever a rewrite fix depends on target capability coverage, regenerate the real repo artifacts and grep the exact reported path pattern in the affected target files before declaring success.
- **Large-extend failures can come from generate-time caches, not import**: I initially treated the Codex remote-extend crash as an import/layout problem, but the cached repo had already imported into `.agentsbridge/skills`; the actual heap blowup came later when `rewriteGeneratedReferences()` rebuilt the full artifact map once per output file. Root cause: I inspected the import surface first and did not measure where the large canonical payload was consumed during generation. Rule: when a big extend triggers OOM, verify whether cached canonical import already succeeded, then audit per-output generation/rewrite caches before changing importer behavior.
- **JSDoc `/*` in block comments**: Using `/*` inside a JSDoc block (e.g. `.agentsbridge/skills/*/SKILL.md`) starts a nested block comment and breaks parsing. Reword to avoid (e.g. `skills/{name}/SKILL.md` or `glob: *.md`).
- **Branch coverage gaps**: To reach 85% branch coverage from ~79%, add tests for: remote-fetcher catch block (offline fallback path), engine.ts uncovered branches, merger/validate-rules, version.ts (both dist/src paths). Unit tests that call runGenerate/runWatch directly (same process) contribute to coverage; integration tests using execSync run in a subprocess and do not.
- **Nested fixture searches**: I incorrectly concluded `root-ab` had no skills because I used `find -maxdepth 3`, which hid `skills/{name}/SKILL.md` and nested reference files. Root cause: bounded search depth on nested fixture structures. Rule: when auditing fixture coverage, use a full-depth listing first or set depth high enough to include nested skill/reference files before drawing conclusions.
- **Canonical skills source**: I added `add-agent-target` only under `.agents/skills/`, which made it a target-specific artifact instead of canonical source data. Root cause: I optimized for local skill discovery and ignored the repo rule that `.agentsbridge/` is the source of truth. Rule: when adding or editing reusable repo skills, put the canonical copy in `.agentsbridge/skills/` first, then generate or align target-specific skill directories from there.
- **Parallel test isolation**: I ran the isolated watch test and the full Vitest suite in parallel even though they use the same fixed temp directory (`ab-watch-cmd-test`). Root cause: I parallelized verification without checking whether the tests shared filesystem state. Rule: never run test commands in parallel when they touch the same temp paths, ports, caches, or repo artifacts.
- **Fixture tree deletion**: I tried to remove the obsolete `root-ab` fixture with a shell `rm -rf`, and the environment blocked it as a destructive command. Root cause: I switched from the repo's patch-based editing workflow to a destructive shell command for tracked files. Rule: when deleting tracked fixture trees in this repo, remove them with `apply_patch` file deletions instead of shell `rm -rf`.
- **Targeted e2e runs use built output**: I ran new e2e slices immediately after patching source and misread failures because `tests/e2e/helpers/run-cli.ts` executes `dist/cli.js`, not source files. Root cause: I forgot that ad hoc `vitest` e2e runs do not build automatically the way `pnpm test:e2e` does. Rule: whenever e2e assertions exercise the CLI binary directly, run `pnpm build` before targeted e2e slices or use `pnpm test:e2e`.
- **Parallel e2e races on dist/**: I ran a targeted e2e slice in parallel with `pnpm test:e2e`, and the full run's `pnpm build` cleaned `dist/` mid-test, causing false CLI "module not found" failures. Root cause: I parallelized verification steps that share the same built output directory. Rule: never run two CLI e2e commands in parallel in this repo; serialize anything that reads `dist/cli.js` or rebuilds `dist/`.
- **CLI integration slices also require fresh dist/**: I ran targeted integration tests after changing CLI behavior and misread a missing generated prompt as an implementation bug, but the cause was stale `dist/cli.js`. Root cause: I remembered the e2e build rule and forgot that integration tests using `node dist/cli.js` have the same dependency. Rule: rebuild with `pnpm build` before any targeted CLI integration or e2e slice whenever source affecting `dist/cli.js` changed.
- **Watch must ignore generated lock artifacts**: Watch mode spammed `Nothing changed`/`Regenerated` in a loop because it observed `.agentsbridge/.lock`, and `generate` rewrites that file atomically on every pass (`.lock.tmp` then `.lock`). Root cause: the watcher subscribed to the whole `.agentsbridge/` tree without excluding its own bookkeeping files. Rule: whenever watch observes canonical config directories, explicitly ignore internal generated artifacts like `.agentsbridge/.lock` and adjacent temp files so self-generated writes cannot retrigger the pipeline.
- **Support-level changes require a full target audit**: I updated the compatibility matrix for embedded agent support but initially left target-specific lint messaging behind, which would have continued warning that the same features were unsupported. Root cause: I treated matrix/docs as the whole support contract and did not audit target diagnostics alongside generator/importer behavior. Rule: whenever a target's support level changes, update and verify the matrix, docs, generator, importer, and target-specific linter messaging together.
- **Remote cache compatibility matters**: I broadened remote extend support and accidentally changed the GitHub cache-key layout, which broke reuse of already-cached GitHub extends and caused refresh tests to look in the wrong directory. Root cause: I generalized the cache naming scheme without preserving the existing GitHub on-disk contract. Rule: when refactoring cache-backed features, preserve existing cache paths for supported providers unless an explicit migration is added and verified.
- **Node globals are not implicit in this lint setup**: I used `URL` directly in new config helpers and lint failed with `no-undef`. Root cause: I assumed Node's runtime global would satisfy the repo's ESLint environment. Rule: when adding Node platform globals in source files here, import them explicitly from the relevant `node:` module instead of relying on ambient globals.
- **Union type refactors need serializer narrowing**: I widened `McpServer` to support URL-based transports, but left the Codex TOML serializer accepting the broad union and only caught it when `pnpm typecheck` failed on `server.command`/`server.args`. Root cause: I updated the source model without re-narrowing downstream sink functions that require a specific variant. Rule: whenever a core type becomes a union, add explicit type guards at every serializer/generator boundary and rerun `pnpm typecheck` before claiming the refactor is done.
- **Reference rewrite audits must check real generated prose, not only exact file refs**: I declared `.agentsbridge/...` reference rewriting fixed, but the generated root instructions still contained skill directory links like `.agentsbridge/skills/post-feature-qa/`. Root cause: my tests covered exact file paths (`SKILL.md`, supporting files) and missed directory-style references used in the actual canonical root rule text. Rule: whenever path rewriting is added or changed, search real canonical content for path patterns and add tests for both exact files and directory-form references before calling the feature complete.
- **Shared artifact collision checks must compare canonicalized rewritten content**: I treated Codex/Windsurf `AGENTS.md` overlaps as ordinary file collisions after adding target-specific reference rewriting, which broke generation even when both outputs came from the same canonical root rule and only differed by rewritten target paths. Root cause: collision handling compared post-rewrite bytes without considering that some overlapping compatibility artifacts intentionally project the same canonical content through different target-specific links. Rule: whenever multiple targets share one output path, add coverage for rewritten-content overlaps and compare the outputs after normalizing target-specific references back to canonical paths before deciding they truly conflict.
- **Fixture-driven e2e requests should use the richest existing fixtures first**: I initially added import-reference e2e coverage with a synthetic inline project even though the user asked for the richest structure fixtures. Root cause: I optimized for speed and forgot that realistic fixture layouts are part of the requirement when verifying path-heavy behavior. Rule: when a user asks for e2e coverage over realistic structures, start from the richest existing fixture and only mutate the copied temp fixture as needed instead of inventing a smaller synthetic project.
- **Import-side reference normalization must not rewrite bare root filenames in prose**: I normalized `AGENTS.md` and similar root artifact names as raw substrings, which changed ordinary prose like `# Windsurf via AGENTS.md` into canonical paths. Root cause: the reverse rewriter treated root filenames as stable path tokens even though they are common standalone words in instructions. Rule: in import-side reference maps, only normalize target paths that uniquely identify canonical artifacts; do not rewrite bare root filenames like `AGENTS.md`, `CLAUDE.md`, `.windsurfrules`, or `GEMINI.md` unless they appear in a more structured path form.
- **Refactor follow-through includes test helper APIs**: I replaced the string-based import reference helper with a file-aware normalizer, but left one unit test file calling the removed function and only caught it during the full `pnpm test` run. Root cause: I updated behavior-facing tests first and forgot to audit direct unit tests that imported the old helper by name. Rule: whenever a core helper API is replaced or removed, run a targeted search for all direct imports/usages and update those tests before relying on the broader suite.
- **Cleanup passes still need typecheck**: I removed projected-agent rebasing bugs, but left unused destructuring behind in two importers and only caught it at the end of `pnpm typecheck`. Root cause: I treated a small cleanup as harmless after tests passed and deferred type-level verification too late. Rule: after any refactor that changes parsing/normalization control flow, rerun `pnpm typecheck` immediately after the code edits, even if the logic change seems minor.
- **Generated-source maps must include every emitted artifact path, not just primary target files**: I mapped Windsurf root rules only to `.windsurfrules`, which left the additional shared `AGENTS.md` outputs outside the generate-side rebasing pass and caused false Codex/Windsurf collisions after import-normalized relative links. Root cause: I treated the reference map as equivalent to the generator's full output set and forgot that one canonical rule can emit multiple artifacts. Rule: whenever a target generates more than one file from the same canonical source, audit the output-source map against the full generator output list so every emitted path participates in rewriting and collision logic.
- **Contract changes must audit “unchanged by design” cases separately from successful rewrites**: I updated the link contract to use project-root paths and initially changed several expectations for references that are intentionally left untouched when the target file does not exist, such as missing checklist refs and the fixture’s invalid `../SKILL.md`. Root cause: I treated all relative-looking references as resolvable without re-checking the existence-based guard in the rewriter. Rule: whenever the link normalization contract changes, explicitly audit both categories: links that should rewrite and links that should stay unchanged because resolution fails.
- **Root-relative link contracts need root-first resolution for bare multi-segment paths**: After switching generated links to project-root-relative paths without a leading slash, I still resolved tokens like `src/AGENTS.md` relative to nested source files, which broke Codex skill imports and only showed up in the rich e2e fixture. Root cause: the resolver only treated known dot-prefixed prefixes as root-relative and assumed other multi-segment paths were source-relative. Rule: when the contract says generated internal links are project-root-relative, resolve bare multi-segment paths from the project root first, then fall back only when the root candidate is not valid, and add nested-file tests for cases like `src/AGENTS.md` and `docs/...`.
- **Multi-output targets need per-output rewrite maps, not just per-target maps**: Copilot emits both `.github/copilot/*.instructions.md` and `.github/instructions/*.md` from the same canonical rule, and I initially cached the rewrite map only by target, which left the new output family using the wrong translation rules and skipping path normalization. Root cause: I assumed one canonical-to-target artifact map was sufficient for every file a target emits. Rule: whenever a target can generate multiple artifact families from the same canonical source, build and cache rewrite maps per output path or per artifact family, and add tests for each emitted path family.
- **Import normalization must map realpath variants of generated artifacts**: I expanded the path rewrite matrix and then left import normalization keying only on joined target paths, which failed in temp-workspace runs where referenced files resolved through a realpath variant and stayed target-specific after import. Root cause: the import artifact map did not canonicalize both direct and realpath forms of generated file paths before translation. Rule: whenever path translation depends on filesystem lookups, include both normalized and realpath variants in the lookup map and add tests that exercise absolute in-project paths through the importer.
- **Path tokenizers must exclude standalone separators used as prose markers**: I treated the leading `/` branch in the shared link regex as “any slash,” which caused status text like `✓ / ✗` to enter the rewrite pipeline and be reported as a missing path. Root cause: the absolute-path branch did not require any actual path segment after `/`. Rule: when broadening path detection, require enough structure to distinguish real file references from prose separators, and add regression tests for marker text like `✓ / ✗`.
- **Cross-platform path fixtures must use valid paths for each file location**: I added Windows-style e2e coverage and briefly used `..\\SKILL.md` from inside `skills/api-generator/template.ts`, which is not a valid sibling reference and caused a false failure that looked like a resolver bug. Root cause: I translated a POSIX relative-path pattern into Windows separators without re-checking the file’s actual directory structure. Rule: when adding cross-platform path tests, validate each fixture path against the real file tree first and only assert normalization on paths that genuinely resolve.
- **URI-skip patterns must not swallow Windows drive letters**: I added protected ranges for external refs like `ssh://...` and `git@github.com:...`, but the first scheme regex also matched `C:` and blocked legitimate Windows absolute paths from being rewritten. Root cause: I treated any single-letter scheme prefix as a URI without separating it from Windows drive syntax. Rule: when protecting URI-like references in this codebase, keep the scheme matcher distinct from drive-letter path detection and add a regression test that covers both in the same file.
- **Real canonical content can contain over-traversal relative refs that still mean repo-root files**: I declared the path rewrite matrix complete, but a checked-in skill still used `../../../../docs/agents-folder-structure-research.md` and generation/import left it unchanged. Root cause: the resolver only tried source-relative normalization for `../` paths and never retried from the repo root when the normalized path escaped above the workspace. Rule: whenever relative-link handling changes, include over-traversal cases from real canonical files in both unit and all-target generate/import matrices, and add a project-root fallback candidate for `../` refs that escape the repo but match an in-repo path.
- **Rewrite-contract changes need a second pass over edge-case suites**: I changed inline-code path rewriting and updated the main unit/e2e coverage first, but left `link-rebaser-edge-cases.test.ts` asserting the old protected-inline behavior and only caught it during the repo-wide `pnpm test` run. Root cause: I audited the primary behavior tests and forgot the separate edge-case suite that codifies the same contract. Rule: whenever the link-rewriter protection contract changes, search both the main rewrite tests and `link-rebaser-edge-cases` for matching expectations before running the full suite.
- **Format-contract work needs the research and round-trip suites, not only e2e slices**: I aligned the generators/importers and got the focused e2e suite green, but `pnpm test` still failed because `tests/agents-folder-structure-research.test.ts` and `tests/import-generate-roundtrip.test.ts` were asserting the legacy file layouts. Root cause: I treated the main e2e matrix as the full contract and forgot the separate research/round-trip suites that also codify official target formats. Rule: whenever a target file-structure contract changes, update and run the research, round-trip, unit, integration, and e2e suites before declaring verification complete.
- **Generated-structure tests must assert exact artifact sets**: Copilot hook coverage passed while still missing the guarantee that every JSON-referenced wrapper script exists, because the tests only checked that some hook files existed. Root cause: the suite used loose structural assertions (`some`, prefix matches, "at least one") instead of exact path lists and exact counts. Rule: whenever a test verifies generated files or directories, assert the exact relative path set and exact count, and cross-check referenced artifacts like hook wrapper scripts against the config that points to them.
- **Parallel generators for one feature must share the same entry filter**: Copilot hook JSON generation treated empty-string commands as valid because it only checked `typeof command === "string"`, while wrapper generation skipped falsy commands and silently produced fewer files. Root cause: two outputs for the same canonical hook set implemented their own “is this hook runnable?” logic. Rule: whenever one canonical feature emits multiple artifact families, centralize the eligibility predicate and add a regression test that compares the referenced artifact list against the emitted files.
- **Target research must separate similarly named products and runtimes**: I answered questions about Codex MCP by mixing the generated `codex-cli` target contract with the current Codex desktop chat runtime, which uses a different configuration surface. Root cause: I treated the target name as equivalent to the active runtime instead of verifying the exact product surface. Rule: when researching or documenting a target, first identify whether the task is about CLI, desktop app, chat product, or repo artifact generation, and verify MCP/config paths for that exact surface before making claims.
- **Exact e2e tree assertions must account for generated lock artifacts**: I added a strict Junie generate e2e assertion and forgot that `generate` writes `.agentsbridge/.lock`, so the otherwise-correct target output failed the exact tree check. Root cause: I copied the canonical tree expectation without auditing the post-generate bookkeeping artifacts that the CLI always emits. Rule: whenever an e2e test asserts the exact generated repo tree, include `.agentsbridge/.lock` if the command under test runs `generate`.
- **Dropped target metadata needs an explicit lint warning**: Codex CLI was silently discarding MCP `description` fields during generation because the target serializer does not project them into `.codex/config.toml`, and I initially explained the behavior without surfacing it through diagnostics. Root cause: I treated unsupported metadata as an implementation detail instead of part of the target contract users need to see during linting. Rule: whenever a target generator intentionally drops canonical metadata, add or update a target-specific lint warning and cover it with unit plus CLI e2e tests.
- **Re-import tests must cover existing canonical metadata, not only empty-project round trips**: Cursor command re-import looked fine in empty-project import tests, but it still stripped `description` and `allowed-tools` when importing generated body-only target files back into an existing `.agentsbridge` tree. Root cause: I tested import completeness and empty-project round trips, but not the real user flow where import overwrites existing canonical files. Rule: whenever a target format cannot represent all canonical metadata, add strict tests that re-import into an existing canonical project and verify unsupported fields are preserved from the canonical destination instead of being erased.
- **Importers must clean stale reserved artifacts when the canonical mapping changes**: After fixing the Codex importer to stop creating hidden/fixture scoped rules and `_ab-*` canonical skill dirs, a live re-import still showed the bad folders/files because the importer only added new outputs and never removed the stale ones from earlier buggy runs. Root cause: I treated import as write-only and forgot that reserved generated names like `_ab-agent-*`, `_ab-command-*`, and skipped scoped-rule names need cleanup on re-import. Rule: when an importer maps reserved/generated source artifacts back into canonical files, add strict re-import tests that preseed stale outputs and verify the importer removes them.
- **Shared importer rules belong in shared code, not copy-pasted per target**: Windsurf had the same hidden/fixture scoped-`AGENTS.md` and stale `_ab-agent-*` cleanup bug right after Codex because both targets implemented parallel importer logic separately. Root cause: I fixed one importer locally instead of extracting the shared contract. Rule: when two targets implement the same import pattern, move the filter/cleanup logic into a shared helper and verify both targets against the same strict regression cases.
- **Import normalization must recover from polluted generated artifacts across target boundaries**: I fixed Codex/Windsurf root imports but still left the repo vulnerable when a different target like `cline` or `claude-code` re-imported a generated root file that already contained Codex-style `.agents/...` links. Root cause: import normalization only loaded rewrite maps for the source target, assuming generated artifacts were pristine and same-family. Rule: when importing any target artifact, build the normalization map from every supported generated path family so re-import can always recover canonical `.agentsbridge/...` links from previously polluted outputs.
- **Coverage matrices must include root-rule references inside non-root files for every target**: I already had broad import/generate rewrite tests, but they still missed Copilot leaving `.github/copilot-instructions.md` inside canonical skill prose because the matrix did not combine exact file-structure assertions with richer cross-file root-rule references for every target on one round trip. Root cause: coverage was split across several suites, each strong locally but not strict enough about the full target contract. Rule: when validating target contracts, add a per-target e2e matrix that checks exact generated files, exact imported canonical files, and representative root/agent/skill link canonicalization on the same generate→import flow.
- **Empty hook commands must be rejected by shared logic, not per-target guesses**: A `Notification` hook with `command: ""` kept reappearing because canonical parsing and Gemini import/generate only checked `typeof command === "string"`, while other paths skipped falsy commands. Root cause: hook eligibility was duplicated across parser, importer, and generator code with different definitions of “valid command”. Rule: centralize trimmed hook command extraction in one helper and use it everywhere hooks are parsed, imported, linted, or generated, with strict tests for empty-string and whitespace-only commands.
- **Canonical booleans should be path-derived and explicit**: Non-root canonical rules drifted between `root: false` and no `root` field at all because each importer serialized rule frontmatter differently. Root cause: the canonical rule contract depended on per-target importer habits instead of one shared serializer. Rule: whenever a canonical field is derivable from the canonical destination path or artifact kind, derive and serialize it in one shared helper and add cross-target tests that assert the exact canonical frontmatter contract.
- **Target feature expansion must update both rewrite path maps and import scanners**: I added Junie commands/agents generation but initially missed reference-map path projection and non-root rule import scanning, which left generated links un-rewritten and round-trip imports incomplete. Root cause: I implemented generator/importer surface files first and forgot the shared path-translation contracts used by rewrite and import normalization. Rule: whenever enabling a new artifact family for a target, update `reference-map-targets` + import-reference map builders + importer scanners together, then verify with target-contract matrix e2e.
- **Shared install-persistence changes need the full entity matrix rerun**: I normalized manual single-item installs to persist their parent collection path plus explicit picks, but initially only updated the command regression and left agent/rule/skill matrix tests asserting the old file-path behavior. Root cause: I verified the user-reported entity in isolation instead of rerunning the broader manual install suites that share the same persistence contract. Rule: whenever install manifest or pack persistence semantics change, rerun and update the full agents/commands/rules/skills matrix, including sibling-item amend cases.
- **Pack naming must describe scope, not whichever item was installed first**: Manual single-item installs were naming packs from the first picked entity, which made sibling-item amendments keep misleading folder names like `repo-code-standards-enforcer` even after the pack grew into a broader collection. Root cause: auto-generated install names reused first-pick naming that works for individual extends entries but not for materialized pack containers. Rule: generate pack names from stable source identity plus normalized install scope, and cover root, scoped, and sibling-amend flows in tests.
- **Stable repo-feature naming requires multi-path provenance, not just a new string format**: I first moved pack names away from first-picked-item naming, but the user wanted all same-repo/same-feature installs to append into one pack, which breaks replay if manifest metadata still stores only one path. Root cause: pack naming and install identity were coupled to a single-path persistence model. Rule: when pack identity is broader than one path, persist aggregated `paths`, match packs by repo identity + feature set, and test sync replay across multiple folders of the same repo/feature.
- **Feature-specific linting must not be nested under unrelated feature gates**: I found that `runLint()` skipped command, MCP, permission, and hook diagnostics whenever `rules` was disabled because the whole per-target loop was gated on `hasRules`. Root cause: lint orchestration mixed per-target dispatch with a single feature guard instead of evaluating each lint family independently. Rule: when a command validates multiple canonical features, gate each lint family by its own feature flag and add a regression test for configs that enable non-rule features without `rules`.
- **Legacy sync replay needs compatibility for umbrella skill folders with nested child skills**: A persisted manual skill install could replay `path: skills/engineering` plus `pick.skills: [release-manager]`, but current staging treated `skills/engineering` as a single skill because it contains its own `SKILL.md`, so sync failed with `No resources match...`. Root cause: manual skill discovery only modeled “single skill” versus “skill collection” from the current folder shape and ignored legacy replay picks that target descendant skills. Rule: when replaying manual skill installs, let saved picked skill names disambiguate umbrella skill containers with nested child skills, and cover the legacy manifest shape with both staging and sync regression tests.
- **Strict install assertions must follow the persisted manifest contract, not assumptions from config or source paths**: While tightening install tests, I initially assumed manual installs would always persist fields like `target`, normalized `path`, and alphabetized picks, which caused new regressions in the hardening pass. Root cause: I wrote stricter expectations from inferred intent instead of checking the actual serialized manifest contract for each install mode. Rule: when converting loose manifest assertions into exact ones, inspect the real persisted entry shape first and lock tests to that observed contract, including field omission and ordering behavior.
- **Replay-scoped installs must carry the narrowed canonical all the way into pack materialization**: I fixed sync replay selection for saved `features`/`pick`, but pack reinstall still wrote unrelated settings like `mcp` and `ignore` because `runInstall()` passed the pre-replay canonical into `installAsPack()`. Root cause: replay narrowing was only applied to pool selection and diagnostics, not to the canonical payload persisted into the pack. Rule: whenever replay or scoped install narrows canonical features, pass that narrowed canonical through every downstream write/materialization path and add a regression test that proves excluded settings are absent from the resulting pack.
- **Manual skill replay must recover picked descendants from collection roots, not only umbrella skill folders**: I fixed nested replay for `path: skills/engineering` but missed the live manifest case `path: skills` with `pick.skills: [release-manager]`, where the selected skill lives under `skills/engineering/release-manager`.
