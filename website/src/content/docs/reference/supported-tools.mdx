AgentsMesh supports every major AI coding tool. This page documents what each tool supports natively, what is embedded/projected, and what is not supported.

> **Plugin-provided targets** do not appear in this matrix — run `agentsmesh plugin list` locally to see which plugin targets are active in your project.

## Legend

| Symbol | Meaning |
|--------|---------|
| **Native** | Feature maps directly to the tool's own config format. Full fidelity. |
| **Embedded** | AgentsMesh projects the feature into a compatible format with metadata for lossless round-trip on import. |
| **Partial** | Limited support — some fields or capabilities may not fully round-trip. |
| **—** | Not supported. Feature is skipped for this target. |

## Global mode

With `--global`, the canonical directory is `~/.agentsmesh/` and CLI commands resolve user-level paths from `$HOME`. **Every** built-in target implements global mode (`descriptor.globalSupport`). The **project** and **global** feature matrices below match `SUPPORT_MATRIX` and `SUPPORT_MATRIX_GLOBAL` in `src/core/matrix/data.ts` (normalized from each target’s capabilities vs `globalSupport.capabilities`). The [Global mode](#global-mode) paths table and each tool section document user-level paths and limitations.

## Tool details

### Amp

**Output files:** `AGENTS.md`, `.agents/skills/`, `.amp/settings.json`

Amp is a coding agent by Sourcegraph. It reads `AGENTS.md` (with fallback to `AGENT.md` or `CLAUDE.md`) for project-level instructions, `.agents/skills/` for skill bundles, and `.amp/settings.json` for MCP servers and workspace settings. There is no dedicated rules directory — non-root rules are embedded in the root file.

**Global mode:** `agentsmesh generate --global` writes to `~/.config/amp/AGENTS.md`, `~/.config/amp/skills/`, and `~/.config/amp/settings.json`. Skills are mirrored to `~/.agents/skills/` unless Codex CLI is also a global target.

| Feature | Format | Notes |
|---------|--------|-------|
| Rules | `AGENTS.md` | Root rule and embedded non-root rules merged into `AGENTS.md`. |
| Commands | — | No native command format; canonical commands are projected as skills via `supportsConversion`. |
| Agents | — | No native agent format; canonical agents are projected as skills via `supportsConversion`. |
| Skills | `.agents/skills/*/SKILL.md` | Skill bundle with `name` and `description`. Shared path with Codex CLI (consumer role). |
| MCP Servers | `.amp/settings.json` | Stored under the `amp.mcpServers` key; existing non-MCP settings are preserved on merge. |
| Hooks | — | Amp has no lifecycle hook system; canonical hooks emit a lint warning. |
| Ignore | — | Amp relies on `.gitignore`; no dedicated ignore file. |
| Permissions | — | Amp permissions are managed via `amp.permissions` in settings.json; canonical permissions emit a lint warning. |

### Claude Code

**Output directory:** `.claude/`

Full native support for all 8 features. The reference implementation for AgentsMesh's canonical format.

| Feature | Format |
|---------|--------|
| Rules | `.claude/rules/*.md` |
| Commands | `.claude/commands/*.md` |
| Agents | `.claude/agents/*.md` |
| Skills | `.claude/skills/*/SKILL.md` |
| MCP Servers | `.claude/settings.json` → `mcpServers` |
| Hooks | `.claude/settings.json` → `hooks` |
| Ignore | `.claudeignore` |
| Permissions | `.claude/settings.json` → `permissions` |

Global mode is fully supported for Claude Code: canonical `~/.agentsmesh/` drives user-level outputs under `~/.claude/`, `~/.claude.json`, and `~/.claudeignore`. Hooks emit to `~/.claude/hooks.json` (project mode keeps hooks in `settings.json`). Skills emit to `~/.claude/skills/` and, when **`codex-cli` is not** among global targets, a compatibility mirror under `~/.agents/skills/` (if Codex is enabled globally it already owns that tree, so Claude does not duplicate there). Agents or commands with frontmatter `outputStyle: true` also emit `~/.claude/output-styles/{name}.md`. Root rules render under `# Global Instructions` in `~/.claude/CLAUDE.md`. See [Global mode](#global-mode) for all targets.

Full native support except permissions (partial).

| Commands | `.cursor/commands/*.md` | |

| MCP Servers | `.cursor/mcp.json` | |
| Hooks | `.cursor/hooks.json` | Cursor uses a standalone hooks file (not `settings.json`) |
| Ignore | `.cursorignore` | |
| Permissions | `.cursor/settings.json` | Partial field coverage |

**Global mode:** writes the same Cursor surfaces as project mode under `~/.cursor/` (per-rule `*.mdc`, `general.mdc`, aggregated `AGENTS.md`, MCP, hooks, ignore, skills, agents, commands). `agentsmesh import --global --from cursor` reads `~/.cursor/rules/*.mdc` first, then falls back to legacy `~/.agentsmesh-exports/cursor/user-rules.md` or `~/.cursor/AGENTS.md`, plus MCP, hooks, ignore, skills, agents, and commands. See `docs/agent-structures/cursor-global-level-generation-strategy.md` for the intended redundancy model.

### Roo Code

**Output directory:** `.roo/`

Roo Code supports native rules, commands, skills, MCP, and ignore. Per-mode rule directories (`.roo/rules-{mode}/`) are imported as regular canonical rules and are not generated back out on generate.

**Global mode:** `agentsmesh generate --global` writes to `~/.roo/rules/*.md`, `~/.roo/commands/*.md`, `~/.roo/skills/`, `~/mcp_settings.json` (Roo’s home-level MCP settings file), `~/.rooignore`, `~/.roo/AGENTS.md` (compatibility aggregate), and `~/.roo/settings/custom_modes.yaml` (canonical agents converted to Roo custom modes). Skills are mirrored to `~/.agents/skills/` unless Codex CLI is also a global target.

| Feature | Format | Notes |
|---------|--------|-------|
| Rules | `.roo/rules/*.md` | Root rule emits as `.roo/rules/00-root.md` (alphabetically first); fallback import from `.roorules` |
| Commands | `.roo/commands/*.md` | `description` frontmatter preserved |
| Agents | `.roomodes` (project), `~/.roo/settings/custom_modes.yaml` (global) | Partial — canonical agents convert to Roo custom modes (`slug`, `name`, `description`, `roleDefinition` from agent body) |
| Skills | `.roo/skills/*/SKILL.md` | Native format with `name`, `description`, `modeSlugs` frontmatter |
| MCP Servers | `.roo/mcp.json` → `mcpServers` | Supports stdio and HTTP/SSE transports |
| Hooks | — | No hook system in Roo Code |
| Ignore | `.rooignore` | gitignore syntax |
| Permissions | — | Not supported |

### Kilo Code

**Output directory:** `.kilo/` plus `AGENTS.md` (root) and `.kilocodeignore`

Kilo Code is a fork of Roo Code (which is a fork of Cline), currently in mid-migration from a legacy `.kilocode/` layout to a new `.kilo/` + `kilo.jsonc` + `AGENTS.md` layout. AgentsMesh **generates** the new layout exclusively; **import** also reads the legacy `.kilocode/` tree and `.kilocodemodes` so existing kilo / Roo-era projects round-trip cleanly.

**Global mode:** `agentsmesh generate --global` writes to `~/.kilo/AGENTS.md`, `~/.kilo/rules/*.md`, `~/.kilo/commands/*.md`, `~/.kilo/agents/*.md`, `~/.kilo/skills/`, `~/.kilo/mcp.json`, and `~/.kilocodeignore`. Skills are mirrored to `~/.agents/skills/` unless Codex CLI is also a global target.

| Feature | Format | Notes |
|---------|--------|-------|
| Rules | `AGENTS.md`, `.kilo/rules/*.md` | Root rule emits as `AGENTS.md` (kilo's documented portable root). Non-root rules carry `description` and `globs` frontmatter when present. |
| Commands | `.kilo/commands/*.md` | `description` frontmatter preserved. Canonical `allowed-tools` is dropped — kilo commands carry only `description`. |
| Agents | `.kilo/agents/*.md` | First-class subagent format (YAML frontmatter `description`, `mode: subagent`, optional `model`). Legacy `.kilocodemodes` `customModes` is imported on round-trip but never generated. |
| Skills | `.kilo/skills/*/SKILL.md` | Anthropic-conventional skill bundle with `name` and `description`. |
| MCP Servers | `.kilo/mcp.json` → `mcpServers` | Imports also fall back to legacy `.kilocode/mcp.json`. |
| Hooks | — | Kilo Code has no user-facing lifecycle hook system; canonical hooks emit a lint warning. |
| Ignore | `.kilocodeignore` | Legacy filename, but the only natively auto-loaded ignore file in kilo today. |
| Permissions | — | Permissions live in `kilo.jsonc`, which agentsmesh does not generate in v1; canonical permissions emit a lint warning. |

### OpenCode

**Output directory:** `.opencode/` plus `AGENTS.md` (root) and `opencode.json` (MCP)

OpenCode is an open-source AI coding agent (CLI/TUI) at [opencode.ai](https://opencode.ai). It supports commands, agents, and skills natively in `.opencode/`. MCP servers are configured in `opencode.json` under the `mcp` key. OpenCode also reads `CLAUDE.md` and `.claude/skills/` as Claude Code compatibility fallbacks, but agentsmesh generates only the native OpenCode paths.

**Global mode:** `agentsmesh generate --global` writes to `~/.config/opencode/AGENTS.md`, `~/.config/opencode/rules/*.md`, `~/.config/opencode/commands/*.md`, `~/.config/opencode/agents/*.md`, `~/.config/opencode/skills/`, and `~/.config/opencode/opencode.json`. Skills are mirrored to `~/.agents/skills/` unless Codex CLI is also a global target.

| Feature | Format | Notes |
|---------|--------|-------|
| Rules | `AGENTS.md`, `.opencode/rules/*.md` | Root rule emits as `AGENTS.md`. Non-root rules carry `description` and `globs` frontmatter. Users should add `".opencode/rules/*.md"` to the `instructions` array in `opencode.json` for OpenCode to read them. |
| Commands | `.opencode/commands/*.md` | `description` frontmatter preserved. Canonical `allowed-tools` is dropped — OpenCode commands carry only `description`. |
| Agents | `.opencode/agents/*.md` | YAML frontmatter with `description`, `mode: subagent`, optional `model`, `tools`, and `disallowedTools`. |
| Skills | `.opencode/skills/*/SKILL.md` | Skill bundle with `name` and `description`. |
| MCP Servers | `opencode.json` → `mcp` | OpenCode uses `mcp` key (not `mcpServers`) with array-style `command` and `environment` instead of `env`. AgentsMesh converts between formats on generate and import. |
| Hooks | — | OpenCode hooks are plugin-based (TypeScript/JavaScript lifecycle events); canonical config hooks emit a lint warning. |
| Ignore | — | OpenCode has no dedicated ignore file. Use `watcher.ignore` in `opencode.json` or `.ignore` (ripgrep) manually. Canonical ignore patterns emit a lint warning. |
| Permissions | — | Permissions live in `opencode.json`, which agentsmesh does not generate in v1; canonical permissions emit a lint warning. |

### Goose

**Output file:** `.goosehints` plus `.agents/skills/` and `.gooseignore`

Goose is an open-source AI coding agent by Block. It reads `.goosehints` (and `AGENTS.md`) for project-level instructions, `.gooseignore` for file exclusion, and `.agents/skills/` for skill bundles. MCP extensions are global-only in `~/.config/goose/config.yaml` and not projected by agentsmesh.

**Global mode:** `agentsmesh generate --global` writes to `~/.config/goose/.goosehints`, `~/.config/goose/.gooseignore`, and `~/.agents/skills/`. Skills are mirrored to `~/.agents/skills/` unless Codex CLI is also a global target.

| Feature | Format | Notes |
|---------|--------|-------|
| Rules | `.goosehints` | Root rule and embedded non-root rules merged into `.goosehints`. |
| Commands | — | Goose slash commands are recipe-based in `config.yaml`; canonical commands emit a lint warning. |
| Agents | — | No first-class agent config files; canonical agents are not projected. |
| Skills | `.agents/skills/*/SKILL.md` | Skill bundle with `name` and `description`. Shared path with Codex CLI. |
| MCP Servers | — | Goose MCP extensions are configured globally in `~/.config/goose/config.yaml`; project-level MCP is not projected. |
| Hooks | — | Goose has no lifecycle hook system; canonical hooks emit a lint warning. |
| Ignore | `.gooseignore` | Gitignore-style patterns. |
| Permissions | — | Goose permissions are runtime-managed via `permission.yaml` in `~/.config/goose/`; canonical permissions emit a lint warning. |

### Warp

**Output files:** `AGENTS.md`, `.warp/skills/`, `.mcp.json`

Warp is an agentic development environment by Warp.dev. It reads `AGENTS.md` (with fallback to legacy `WARP.md`) for project-level instructions, `.warp/skills/` for skill bundles, and `.mcp.json` (standard MCP format) for MCP servers. There is no dedicated rules directory — non-root rules are embedded in the root file. When both `WARP.md` and `AGENTS.md` exist, `WARP.md` takes priority. Warp also scans `.agents/skills/`, `.claude/skills/`, and other tool skill directories.

**Global mode:** `agentsmesh generate --global` writes skills to `~/.warp/skills/`. Warp's global rules are managed via Warp Drive (UI), not file-based — global mode is limited to skills.

| Feature | Format | Notes |
|---------|--------|-------|
| Rules | `AGENTS.md` | Root rule and embedded non-root rules merged into `AGENTS.md`. Legacy `WARP.md` supported on import. |
| Commands | — | No native command format; canonical commands are projected as skills via `supportsConversion`. |
| Agents | — | No native agent format; canonical agents are projected as skills via `supportsConversion`. |
| Skills | `.warp/skills/*/SKILL.md` | Skill bundle with `name` and `description` frontmatter. |
| MCP Servers | `.mcp.json` | Standard `mcpServers` format at project root; shared with Claude Code. |
| Hooks | — | Warp has no lifecycle hook system; canonical hooks emit a lint warning. |
| Ignore | — | Warp relies on `.gitignore`; no dedicated ignore file. |
| Permissions | — | Warp permissions are UI-managed; canonical permissions emit a lint warning. |

### Zed

**Output files:** `.rules`, `.zed/settings.json`

Zed is a modern code editor with a built-in AI assistant (zed.dev). It reads `.rules` for project-level AI instructions and `.zed/settings.json` for MCP servers (`context_servers` key). Zed also reads `.cursorrules`, `AGENTS.md`, `CLAUDE.md`, and `GEMINI.md` as fallbacks, but agentsmesh generates to the native `.rules` path. There is no dedicated rules directory — non-root rules are embedded in the single `.rules` file. Zed has no native commands, agents, skills, hooks, ignore, or permissions file formats.

**Global mode:** `agentsmesh generate --global` writes MCP configuration to `~/.config/zed/settings.json`. Zed has no global rules file — global mode is limited to MCP server configuration.

| Feature | Format | Notes |
|---------|--------|-------|
| Rules | `.rules` | Root rule and embedded non-root rules merged into `.rules`. |
| Commands | — | No native command format; not projected. |
| Agents | — | No native agent format; not projected. |
| Skills | — | No native skill format; not projected. |
| MCP Servers | `.zed/settings.json` | Stored under the `context_servers` key; existing non-MCP settings are preserved on merge. |
| Hooks | — | Zed has no lifecycle hook system; canonical hooks emit a lint warning. |
| Ignore | — | Zed relies on `file_scan_exclusions` in settings.json; no dedicated ignore file. |
| Permissions | — | Zed permissions are managed via `agent.tool_permissions` in settings.json; canonical permissions emit a lint warning. |
